# Validation Alone vs Validation Together 비교 분석

## 데이터 요약

### Validation_alone (단독 실행)
- 각 모델이 순차적으로 단독 실행
- GPU 리소스를 독점적으로 사용
- 정확도 메트릭 계산 포함

### Validation_together (함께 실행)
- 5개 모델이 동시에 실행
- GPU 리소스 공유
- 정확도 메트릭 계산 포함

## 주요 경향성 분석

### 1. 추론 시간 성능 저하

#### 심각한 성능 저하 모델
- **yolo11n-detect**: 74.6% 느려짐 (27.5ms → 48.1ms) ⚠️
  - 단독 실행: 27.5ms
  - 함께 실행: 48.1ms
  - 가장 큰 성능 저하를 보임

#### 중간 수준 성능 저하 모델
- **yolo11n-cls**: 29.1% 느려짐 (12.3ms → 15.8ms)
  - 경량 모델이지만 함께 실행 시 영향이 큼
- **yolo11n-seg**: 17.2% 느려짐 (37.3ms → 43.8ms)
- **yolo11n-pose**: 13.0% 느려짐 (29.1ms → 32.9ms)

#### 낮은 수준 성능 저하 모델
- **yolo11n-obb**: 6.4% 느려짐 (57.3ms → 61.0ms)
  - 이미 가장 느린 모델이라 상대적 영향이 적음

### 2. 정확도 메트릭

**중요 발견**: 모든 모델에서 정확도 메트릭이 **완전히 동일**합니다.
- Classification: Top-1, Top-5 동일
- Detection: Precision, Recall, F1 동일
- Pose: Precision, Recall, F1 동일
- Segmentation: Precision, Recall, F1 동일
- OBB: Detection 수 동일

**분석**: 함께 실행 시 추론 시간은 느려지지만, **모델의 정확도는 전혀 영향받지 않습니다**. 이는 GPU 리소스 경쟁이 추론 속도에만 영향을 미치고, 모델의 추론 결과 자체는 변하지 않음을 의미합니다.

### 3. 모델별 특성

#### 경량 모델 (yolo11n-cls)
- **특징**: 가장 빠른 추론 시간 (~12ms)
- **경향**: 함께 실행 시 29.1% 성능 저하
- **이유**: 경량이라 상대적 영향이 큼

#### 중간 복잡도 모델 (yolo11n-detect, yolo11n-pose, yolo11n-seg)
- **특징**: 중간 수준의 추론 시간 (27-38ms)
- **경향**: detect는 74.6% 저하, pose는 13.0% 저하, seg는 17.2% 저하
- **이유**: 모델별 GPU 연산 패턴 차이로 인한 리소스 경쟁 영향 차이

#### 고복잡도 모델 (yolo11n-obb)
- **특징**: 가장 느린 추론 시간 (~57ms)
- **경향**: 함께 실행 시 6.4% 저하로 가장 적은 영향
- **이유**: 이미 연산량이 많아 상대적 영향이 적음

### 4. Validation_alone 내부 변동성

두 번의 실행 결과를 비교하면:
- **yolo11n-cls**: 5.2% 차이 (11.963ms vs 12.580ms)
- **yolo11n-detect**: 5.8% 차이 (26.752ms vs 28.316ms)
- **yolo11n-pose**: 7.5% 차이 (30.263ms vs 28.002ms) - 오히려 빨라짐
- **yolo11n-seg**: 6.5% 차이 (36.157ms vs 38.491ms)
- **yolo11n-obb**: 2.5% 차이 (56.611ms vs 58.035ms)

**분석**: 단독 실행에서도 약간의 변동성이 있으며, 이는 시스템 상태, GPU 온도, 메모리 상태 등에 의한 것으로 보입니다.

### 5. Validation_together 내부 변동성

두 번의 실행 결과를 비교하면:
- **yolo11n-cls**: 0.2% 차이 (15.827ms vs 15.856ms)
- **yolo11n-detect**: 1.4% 차이 (48.416ms vs 47.736ms)
- **yolo11n-pose**: 0.4% 차이 (33.004ms vs 32.866ms)
- **yolo11n-seg**: 11.4% 차이 (46.415ms vs 41.107ms) - 가장 큰 변동
- **yolo11n-obb**: 5.6% 차이 (62.764ms vs 59.270ms)

**분석**: 함께 실행 시 seg 모델에서 가장 큰 변동성이 있으며, 이는 동시 실행 시 리소스 할당의 불확실성 때문으로 보입니다.

## 결론 및 권장사항

### 1. 성능 vs 정확도 트레이드오프

- **추론 시간**: 함께 실행 시 모든 모델에서 성능 저하 발생
  - detect: 74.6% 저하 (가장 심각)
  - cls: 29.1% 저하
  - seg: 17.2% 저하
  - pose: 13.0% 저하
  - obb: 6.4% 저하 (가장 적음)

- **정확도**: 모든 모델에서 **완전히 동일**
  - 함께 실행해도 모델의 정확도는 전혀 영향받지 않음
  - 이는 매우 중요한 발견

### 2. 실무 적용 시나리오

#### 단독 실행 권장 상황
- **실시간 추론이 중요한 경우**
- **latency가 중요한 애플리케이션**
- **개별 모델의 최적 성능이 필요한 경우**

#### 함께 실행 고려 상황
- **배치 처리 시**: 전체 처리 시간 단축 가능
- **정확도는 유지하면서 처리량을 높이고 싶은 경우**
- **리소스가 제한적인 환경에서 여러 모델을 동시에 사용해야 하는 경우**

### 3. 모델별 우선순위

1. **yolo11n-obb**: 함께 실행 영향이 가장 적음 (6.4%)
2. **yolo11n-pose**: 함께 실행 영향이 적음 (13.0%)
3. **yolo11n-seg**: 함께 실행 영향이 중간 (17.2%)
4. **yolo11n-cls**: 함께 실행 영향이 큼 (29.1%)
5. **yolo11n-detect**: 함께 실행 영향이 매우 큼 (74.6%) ⚠️

### 4. 최종 권장사항

- **정확도가 최우선인 경우**: 함께 실행해도 정확도는 동일하므로 고려 가능
- **성능이 최우선인 경우**: 단독 실행 권장
- **균형잡힌 접근**: 중요도가 낮은 모델은 함께 실행, 중요도가 높은 모델은 단독 실행
- **yolo11n-detect 사용 시**: 함께 실행 시 성능 저하가 매우 크므로 단독 실행 강력 권장
